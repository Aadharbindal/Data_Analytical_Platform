1. Objective
This chapter defines how business data, analytical metadata, conversations, reports and AI-generated insights are stored. The architecture separates raw datasets from processed analytical artifacts to improve scalability, traceability and future extensibility.

5. Table Design Guidelines
Use UUID primary keys for externally exposed entities.
Include created_at and updated_at timestamps.
Support soft deletion where business recovery is important.
Store analytical artifacts separately from raw uploads.
Maintain foreign key integrity between workspaces and assets.
6. Indexing Strategy
Index workspace_id on analytical tables.
Index dataset_id for KPI, insight and chart lookups.
Full-text index on conversation messages (future).
Composite indexes for common dashboard queries.
7. Versioning & Metadata
Each uploaded dataset receives a unique version identifier. Profiles, KPIs and insights are linked to the dataset version to ensure reproducibility. Re-uploading a modified file creates a new version rather than overwriting historical analysis.
8. Backup & Recovery
Metadata should be backed up through scheduled PostgreSQL backups. Raw datasets and exported reports should be replicated through object storage policies. Recovery procedures must preserve referential integrity between metadata and files.
9. Data Lifecycle
Upload → Registration → Profiling → Analytics → Insight Generation → Reporting → Archive → Retention/Deletion.Retention policies should be configurable to support enterprise governance requirements.
